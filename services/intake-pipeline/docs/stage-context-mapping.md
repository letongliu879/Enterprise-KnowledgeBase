# StageContext -> Target Architecture Field Mapping

> 规范性状态：历史迁移说明，非最终架构入口。
>
> 本文用于解释旧 `StageContext` 字段如何映射到最终态架构，不单独定义当前实现边界。最终边界以 `docs/上游机制移植总览.md`、`services/intake-pipeline/intake-pipeline.md`、`services/indexing/indexing.md`、`services/retrieval/retrieval.md` 为准。

Historical baseline document. Extracted from `test_stages_snapshot.py` inline
comments and expanded with migration targets per phase.

This document answers two questions:
1. What in the current `StageContext` maps to what in the target architecture?
2. Which fields are transitional (to be removed/migrated) and which must be
   retained through to the target architecture?

## Target fields NOT present in current `StageContext`

These fields exist in `intake-pipeline.md` target architecture but have no
corresponding field in the current `StageContext`.  They are introduced in
later phases.

| Target field | Introduced in | Where it lives in target architecture |
|--------------|---------------|---------------------------------------|
| `publish_id` | Phase 6 | `publish_jobs.publish_id`; created by publishing-worker |
| `active_index_version` | Phase 6 | `published_documents.active_index_version`; set after `IndexReady` |
| `published_document_state` | Phase 6 | `published_documents.state` (`PUBLISHED/ARCHIVED/DEPRECATED/RETRACTED/REINDEXING`) |
| `supersedes_final_doc_id` | Phase 6 | `published_documents.supersedes_final_doc_id`; set on version conflict resolution |
| `created_by_ticket_id` | Phase 5 | `published_documents.created_by_ticket_id`; audit trace from approval to publish |
| `reindex_job_id` | Phase 6 | `reindex_jobs.reindex_job_id`; separate from `publish_id` |
| `index_build_job_id` | Phase 6 | `index_build_jobs.index_build_job_id`; indexing-service internal task |
| `chunk_count` / `embedding_count` | Phase 6 | `index_build_jobs`; indexing-service progress tracking |
| `stage_task_id` | Phase 2 | `stage_tasks.stage_task_id`; already in orchestrator but not in context |
| `stage_attempt_id` | Phase 2 | `stage_attempts.stage_attempt_id`; already in orchestrator but not in context |
| `idempotency_key` | Phase 2 | `stage_tasks.idempotency_key`; computed from `intake_job_id:stage_name:schema_version:input_hash` |
| `rerun_round` / `return_reason_code` | Phase 5 | `stage_tasks`; set on approval return retry |

## Migration summary by phase

| Phase | What changes in `StageContext` |
|-------|-------------------------------|
| Phase 1 | `doc_id` -> `preliminary_doc_id` in schemas; pure executors no longer need `session` |
| Phase 2 | `job_id` coexists with `intake_job_id`; orchestrator fields (`stage_task_id`, `attempt_id`) recorded externally |
| Phase 3 | `collection`, `tenant`, `document_repo`, `policy_repo` removed; stages receive primitives + domain services |
| Phase 4 | `source_file_path` becomes legacy; `source_file_id` + `object_id` become primary references |
| Phase 5 | `publish_status` source moves to approval-service; `final_doc_id`, `ticket_id` added |
| Phase 6 | `asset_bundle` moves to `IndexBuildInput`; `publish_id`, `active_index_version` introduced in new tables |
| Phase 9 | `monitor` replaced by OpenTelemetry; `review_context` may drop from business schema |

## Notes

- This document is append-only during Phase 0.  Subsequent phases must update
  it when fields are migrated or removed, but must not delete historical rows.
- Fields marked [TRANSITIONAL] must have a test in `test_stages_snapshot.py`
  that documents their current behavior and asserts it will change in the
  target phase.
- Fields marked [RETAIN] must have a test that asserts they survive unchanged
  through to the target architecture.
