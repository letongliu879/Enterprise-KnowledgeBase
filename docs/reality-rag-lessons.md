# 从 Reality-RAG 继承什么，不继承什么

## 1. 这份文档的用途

`Reality-RAG` 对当前项目仍然有价值，但它的价值不是“恢复旧代码形态”，而是：

- 提供边界判断
- 提供失败经验
- 提供可复用的方法论

因此，当前项目应该把 `Reality-RAG` 视为：

- 架构原则来源
- 上游吸收策略来源
- 反模式清单来源

而不是目标实现蓝图。

## 2. 应该继承的部分

### 2.1 平台拥有治理真相

这一条必须原封不动继承：

- 文档身份由平台定义
- collection 归属由平台定义
- ACL 由平台定义
- 审批和发布由平台定义
- 生命周期由平台定义

`RAGFlow` 只能做解析和工作台，不应该成为真相源。

### 2.2 Java retrieval / Python intake 的职责分工

旧项目最清楚的一点，就是在线和离线能力分工：

- Python 适合摄入、审批、发布、索引构建
- Java 适合在线接入、在线检索、MCP / REST 暴露

这个分工在当前项目依然成立。

### 2.3 契约优先

旧项目反复强调：

- `contracts/` 应是唯一跨语言契约来源
- 本地 DTO 只能做镜像或过渡
- 不能各服务各自演进“差不多”的契约

这条在当前项目里仍然应该保持高优先级。

### 2.4 检索不是只做召回

旧项目对 retrieval 的理解是对的：

- 权限先于检索
- hybrid recall 是主线
- rerank 只是中段
- 真正的价值在 context engineering

也就是：

- smart cutoff
- adjacent expansion
- section / breadcrumb expansion
- token budget pack

这些来自 `ContextWeaver` 的思想仍然值得保留。

## 3. 不应该继续继承的部分

### 3.1 “移植上游能力”这套叙事

旧项目最大的问题之一，不是边界判断错了，而是太容易把工作重心转成：

- 迁移多少上游模块
- 对齐多少目录结构
- 做多少 adapter

这会让项目不断围着上游转，而不是围着平台主线转。

当前项目应该改成：

- 先定义平台边界
- 再决定哪些能力直接复用
- 最后决定哪些地方需要适配层

### 3.2 把成熟上游模块拆碎再重写

旧项目文档里最有价值的一条提醒就是：

- 能整块复用的，不要先拆碎

因为一旦拆碎，最后通常只会留下：

- 名字像
- 步骤像
- 行为细节却丢了

这对 `RAGFlow` 的解析/分块、对 `ContextWeaver` 的上下文工程尤其危险。

### 3.3 让 workbench 对象污染治理模型

旧项目已经清楚区分了：

- `dataset/file` 是工作台对象
- `collection/final_doc_id` 是平台对象

当前项目如果再次把两者混在一起，后面 ACL、发布、索引切换、检索过滤都会变乱。

### 3.4 过度保留旧包袱

旧项目里很多内容本质上属于过渡产物：

- adapter 层
- 历史包结构
- 兼容性镜像
- 为迁移服务的临时组织方式

这些不应该被继续升格为当前项目主线。

## 4. 旧文档应该怎么读

### 4.1 可以当原则文档读的

- `docs/最终主线.md`
  - 用来理解服务边界、主链和全局立场
- `docs/上游机制移植总览.md`
  - 用来理解“整块复用 / sidecar / 接口借形状 / 明确不搬”这套判断框架
- `docs/RAGFlow策略分析.md`
  - 用来理解为什么 `RAGFlow` 适合承接解析、分块、工作台
- `docs/ContextWeaver策略分析.md`
  - 用来理解哪些上下文工程能力值得吸收

### 4.2 不能直接当当前实现说明读的

- 任何依赖旧目录结构的描述
- 任何把“迁移计划”当“当前状态”的描述
- 任何围绕旧 adapter / legacy 组织的实现路线

## 5. 对当前项目的直接启发

### 5.1 `upstream/ragflow` 应该被当作受控引擎

这意味着：

- 保留对解析、分块、工作台真正有价值的模块
- 裁掉 agent/chat/memory/mcp 等产品边界
- 允许为 dataflow 保留必要依赖
- 但不允许它反客为主成为平台真相源

### 5.2 `services/retrieval` 应该优先做实

因为当前仓库里，Java retrieval 已经比很多其他模块更接近“真实主线”：

- 有代码
- 有配置
- 有测试
- 有 RAGFlow / ContextWeaver 吸收点

它比一些还停留在目录层面的模块更值得优先固化。

### 5.3 `workbench-api` 和 `ragflow-adapter` 不能靠名字判断成熟度

当前仓库里，这两个目录都不应直接当成稳定边界依据。

后续需要：

- 明确它们是否真的承担主线职责
- 如果承担，就补齐可维护源码和契约
- 如果不承担，就不要让它们继续制造叙事噪音

## 6. 当前项目应避免重蹈的反模式

- 不要再写“为了以后迁移方便”的抽象层，然后长期无人消费。
- 不要把上游整套产品边界搬进来，再试图慢慢删。
- 不要让旧文档里的阶段性说法继续充当当前架构真相。
- 不要在没有主线闭环前，先花大量精力恢复外壳目录。
- 不要把能直接复用的成熟模块改写成一组表面相似的本地实现。

## 7. 一句话

`Reality-RAG` 最值得继承的是边界判断，不是迁移姿势；最该吸取的教训，是不要再让“接上游”这件事本身，压过“建立当前项目自己的主线”。
