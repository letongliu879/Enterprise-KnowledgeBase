# 99 - 端到端核心流程验收

> **文档定位**：本文件验证企业知识库工作台最核心的端到端业务链路是否畅通。它不替代各模块文档，而是把各模块串成一条“用户真实工作流”，确保单点可用 ≠ 链路断节。
>
> **覆盖流程**：选择集合 → 批量上传 → 解析复核 → 审批决策 → 文档入库 → 检索验证。
>
> **前置依赖**：必须先完成 `00-全局可用性基线与验收规则.md` 的基线验收。
>
> **关联需求**：`需求.md` 第 3、4、5、6、7、8 章。

---

## 1. 验收目标与范围

### 1.1 业务目标

验证一个标准操作员能否独立完成以下完整工作流：

1. 选择当前操作的知识库集合；
2. 上传一批文档到该集合；
3. 在复核队列中找到待复核的工单；
4. 打开复核详情，查看源文件、解析 chunks、Agent 审核发现；
5. 做出审批决策（通过 / 驳回 / 退回修改）；
6. 审批通过的文档进入文档库并可被检索；
7. 在检索验证页查询到相关证据片段。

### 1.2 涉及页面

| 路由 | 页面 | 作用 |
|---|---|---|
| `/collections` | 知识库集合页 | 选择/创建集合 |
| `/upload` | 批量入库页 | 上传文件并追踪状态 |
| `/review` | 人工复核队列页 | 查看待复核工单 |
| `/review/[taskId]` | 复核详情页 | 人工审批决策 |
| `/documents` | 文档库页 | 查看已发布文档 |
| `/retrieval` | 检索验证页 | 验证检索效果 |

### 1.3 涉及角色

- **操作员**：上传文件、提交复核、执行检索；
- **复核员**：在复核详情页审批工单；
- **管理员**：处理批量操作、生命周期管理（本流程中可选）。

## 2. 用户角色、前置条件与可用性基线

### 2.1 角色与权限

| 角色 | 权限 | 本流程中使用场景 |
|---|---|---|
| 操作员 | 上传文件、查看任务、执行检索 | 上传、检索 |
| 复核员 | 查看工单详情、Approve/Reject/Return | 复核决策 |
| 管理员 | 批量操作、归档/撤回/重建索引 | 文档库高级操作（可选） |

### 2.2 前置条件

- 已配置有效的 `demoToken`（JWT）和 `demoApiKey`；
- 后端服务 `admin`、`workbench`、`access`、`retrieval`、`ingestion` 至少部分健康；
- 存在至少一个可访问的知识库集合（若无，需先在 `/collections` 创建）；
- 测试文件准备：至少 1 个 PDF 或 DOCX（大小 < 50MB，格式在支持范围内）。

### 2.3 可用性基线

本文档默认遵循 `00-全局可用性基线与验收规则.md`。核心基线重申：

- 每个页面切换必须有加载反馈；
- 每个操作必须有成功/失败 Toast；
- 后端 501 必须展示 BackendGap；
- 空列表必须展示 EmptyState。

## 3. 全链路场景叙事（A - 主结构）

### 场景 1：标准文档入库与检索验证流程

**用户**：操作员 + 复核员
**目标**：完成从上传、复核、发布到检索验证的完整链路。
**优先级**：P0

#### 步骤 1：选择知识库集合

**操作描述**：
1. 操作员进入 `/collections` 页面；
2. 在集合卡片列表中，找到目标集合；
3. 点击该卡片上的“选择用于上传”按钮。

**预期界面状态**：
- 页面加载时展示 3 个骨架屏卡片；
- 集合列表加载完成后，每张卡片显示文件夹图标、集合名称、生命周期状态 Badge、集合 ID、描述、租户 ID；
- 点击“选择用于上传”后，顶部 Header 的集合选择器同步更新为当前集合；
- Toast 提示“已选择集合：XXX”。

**[B] 单点验收点**：
- `E2E-COL-001`：集合选择后，Zustand 中 `currentCollectionId` 被正确设置；
- `E2E-COL-002`：Header 集合选择器显示当前选中集合名称；
- `E2E-COL-003`：未选择集合时进入 `/upload`，上传按钮 disabled 并提示错误。

**[C] 视觉规格**：
- 集合卡片采用 1-3 列响应式网格；
- 选中/悬停时有渐变覆盖效果；
- 生命周期状态 Badge 颜色正确（active 用 success 绿色）。

#### 步骤 2：上传文件

**操作描述**：
1. 操作员进入 `/upload`；
2. 将测试文件拖拽到上传区，或点击上传区唤起文件选择器；
3. 观察文件卡片出现并经历状态流转；
4. 等待文件进入“复核中”状态。

**预期界面状态**：
- 拖拽进入上传区时，边框高亮 + 渐变背景动画；
- 未选择集合时，上传区禁用并 Toast 提示“请先选择集合”；
- 文件选择后，文件卡片按顺序加入列表，显示文件类型图标、文件名、大小、状态徽标；
- 状态流转：排队 → 上传中 → 已上传 → 解析中 → 复核中；
- 上传中/解析中/索引构建中卡片底部显示 shimmer 进度条；
- 统计面板显示总数、处理中、已入库、待复核、失败 5 个指标；
- 最近任务列表按上传时间倒序展示，当前文件状态与任务状态联动更新。

**[B] 单点验收点**：
- `E2E-UPL-001`：支持 PDF/DOCX/PPTX/XLSX/CSV 格式；
- `E2E-UPL-002`：最大 3 个并发上传，其余排队；
- `E2E-UPL-003`：未选择集合或未配置权限范围时禁止上传；
- `E2E-UPL-004`：文件状态通过轮询（5s/30s）正确更新；
- `E2E-UPL-005`：上传失败时展示错误信息 + 重试按钮。

**[C] 视觉规格**：
- 文件卡片左侧状态色条随状态变色；
- 上传区拖拽高亮时边框颜色变化明显；
- 统计卡片为 5 宫格布局，数字清晰可读。

#### 步骤 3：进入复核队列

**操作描述**：
1. 操作员点击侧边栏“人工复核”进入 `/review`；
2. 在队列中查看刚上传文件生成的工单；
3. 点击工单卡片进入 `/review/[taskId]`。

**预期界面状态**：
- 工单列表按更新时间倒序展示；
- 每条工单卡片显示状态色条、状态图标、文件名、集合 ID Badge、工单 ID（monospace）、相对时间、状态徽标；
- 待复核状态徽标带脉冲动画；
- 卡片右侧有 Chevron 导航指示；
- 顶部筛选器可按集合、状态筛选；
- 存在清除筛选按钮（仅在筛选非默认时展示）。

**[B] 单点验收点**：
- `E2E-REV-001`：刚上传的文件在队列中出现，状态为“待复核”；
- `E2E-REV-002`：点击卡片正确跳转 `/review/[taskId]`；
- `E2E-REV-003`：筛选器按集合/状态过滤结果正确；
- `E2E-REV-004`：无工单时展示 EmptyState。

**[C] 视觉规格**：
- 待复核脉冲动画明显但不刺眼；
- 工单 ID 使用等宽字体；
- 卡片 hover 时有轻微位移阴影。

#### 步骤 4：复核详情审查

**操作描述**：
1. 复核员在 `/review/[taskId]` 查看工单详情；
2. 查看头部 Hero Section 的四宫格指标和元数据网格；
3. 切换 Source / Draft edits / Agent review 三个 Tab；
4. 在 Source Tab 查看 PDF/图片/HTML/文本预览；
5. 在 Draft edits Tab 查看 chunks 列表；
6. 在 Agent review Tab 查看 findings 列表；
7. 若 workspace 部分数据不可用，查看 Alert 提示。

**预期界面状态**：
- 头部显示返回按钮、状态徽标组、文档标题、描述、四宫格指标卡片（Findings/Risks/Draft edits/Warnings）、元数据网格；
- Source Tab 中 DocumentViewer 正确渲染源文件，工具栏有缩放、复制、下载按钮，分页导航可用；
- Draft edits Tab 中 ChunkEditorWorkbench 展示 evidence_id Badge、section_path、页码范围，长内容可展开折叠；
- Agent review Tab 中 findings 按 Critical → High → Medium → Low → Info 排序；
- 每条 Finding 展开后显示 Source quote、Find in document 按钮、Edit chunk 按钮、页码范围；
- 右侧边栏展示 Decision Cockpit、Ticket Context、System & Diagnostics。

**[B] 单点验收点**：
- `E2E-RVD-001`：三个 Tab 可正常切换，当前 Tab 有高亮指示；
- `E2E-RVD-002`：Source Tab 支持 PDF/图片/HTML/文本；
- `E2E-RVD-003`：Agent review findings 按严重程度排序；
- `E2E-RVD-004`：从 Finding 点击“Find in document”可自动搜索高亮定位；
- `E2E-RVD-005`：从 Finding 点击“Edit chunk”可自动滚动到对应 chunk；
- `E2E-RVD-006`：degraded_parts 存在时展示 Alert。

**[C] 视觉规格**：
- 四宫格指标卡片布局整齐，数字突出；
- Tab 切换带 layoutId 弹簧动画指示器；
- Severity Badge 颜色正确（Critical 红色、High 橙色、Medium 黄色、Low 蓝色、Info 灰色）。

#### 步骤 5：做出审批决策

**操作描述**：
1. 复核员在右侧 Decision Cockpit 中查看当前状态；
2. 若工单处于待复核状态且有权限，在决策原因文本框中输入原因（可选）；
3. 点击 Approve（通过）、Reject（驳回）或 Return for revision（退回修改）之一。

**预期界面状态**：
- 待复核状态显示 Approve / Reject / Return 三个按钮；
- 点击后弹出确认 Dialog，要求输入/确认原因；
- 决策成功后 Toast 提示，工单状态更新；
- 决策原因展示在 Decision Cockpit 中；
- 若无权限，操作按钮不可见或 disabled。

**[B] 单点验收点**：
- `E2E-DEC-001`：Approve 后工单状态变为“已批准”；
- `E2E-DEC-002`：Reject 后工单状态变为“已拒绝”；
- `E2E-DEC-003`：Return 后工单状态变为“已退回”；
- `E2E-DEC-004`：无权限用户看不到决策按钮；
- `E2E-DEC-005`：决策原因可输入并在决策后展示。

**[C] 视觉规格**：
- Approve 按钮使用 primary/blue 或 success 绿色；
- Reject / Return 使用 destructive 红色或 secondary；
- 确认 Dialog 为 glass 风格圆角弹窗。

#### 步骤 6：文档进入文档库

**操作描述**：
1. 审批通过后，操作员进入 `/documents`；
2. 在文档列表中找到刚发布的文档；
3. 点击文件名进入 `/documents/[docId]`。

**预期界面状态**：
- 文档列表按 `projection_updated_at` 倒序展示，最多 200 条；
- 每条文档卡片显示复选框（管理员）、文件类型图标、文件名、Review 状态 Badge、Indexed Badge、集合 ID、chunk 数、页数、完整度、相对更新时间、Stale 标记、状态徽标；
- 审批通过的文档 Review 状态为 Approved，Indexed Badge 为已索引；
- 点击文件名跳转 `/documents/[docId]`。

**[B] 单点验收点**：
- `E2E-DOC-001`：Approve 后文档出现在文档库；
- `E2E-DOC-002`：文档状态、Review 状态、Indexed 状态正确；
- `E2E-DOC-003`：文档详情页头部展示正确状态徽标组；
- `E2E-DOC-004`：文档详情 Source / Chunks / Agent 三个 Tab 与复核详情功能一致。

**[C] 视觉规格**：
- 文件类型图标颜色按扩展名区分（PDF 红色、DOC 蓝色、PPT 橙色、Sheet 绿色）；
- Stale 标记使用 destructive 红色 Badge；
- Indexed Badge 使用 success 绿色。

#### 步骤 7：检索验证

**操作描述**：
1. 操作员进入 `/retrieval`；
2. 输入与刚发布文档相关的查询词；
3. 设置 Token 预算；
4. 选择集合和检索配置；
5. 点击检索按钮；
6. 查看证据片段列表。

**预期界面状态**：
- 未选择集合或配置时，检索按钮 disabled 并展示 Alert 提示；
- 检索成功后展示证据片段列表；
- 结果头部显示标题、展开全部/收起全部按钮、计数 Badge（X 项 · 已用 Y Token）；
- 每条证据卡片显示序号 Badge（渐变背景）、相关性分数 + 进度条、复制按钮、展开/折叠按钮、元数据行、内容区；
- 展开后展示 source_stage、why_selected、section_path；
- 空结果时展示 EmptyState。

**[B] 单点验收点**：
- `E2E-RET-001`：未选集合/配置时禁用检索；
- `E2E-RET-002`：检索 API 返回 `evidence_items`；
- `E2E-RET-003`：每条 evidence_item 显示 score、doc_id、evidence_id、collection_id；
- `E2E-RET-004`：复制内容按钮有 1.5s 成功状态反馈；
- `E2E-RET-005`：Token 使用量正确显示。

**[C] 视觉规格**：
- 序号 Badge 使用渐变背景；
- 相关性分数进度条颜色随分数变化；
- 检索按钮禁用状态明显。

## 4. 异常 / 容错场景

### 场景 2：上传后解析失败

**优先级**：P0
**异常点**：文件上传成功但解析失败。

**预期系统行为**：
- 文件卡片状态变为“失败”，显示错误信息；
- 提供重试按钮；
- 统计面板“失败”计数 +1；
- 不生成复核工单。

**验收点**：
- `E2E-ERR-001`：解析失败的文件展示错误原因；
- `E2E-ERR-002`：点击重试后重新进入解析流程；
- `E2E-ERR-003`：失败状态颜色为 destructive 红色。

### 场景 3：复核时后端服务离线

**优先级**：P1
**异常点**：进入 `/review/[taskId]` 时 workbench-api 不可用。

**预期系统行为**：
- 页面不白屏；
- 展示错误 Alert 或 BackendGap；
- 提供重试入口；
- 已加载的静态 UI（如侧边栏、Header）保持可用。

**验收点**：
- `E2E-ERR-004`：API 超时/离线时展示友好错误；
- `E2E-ERR-005`：错误页面包含刷新/重试按钮。

### 场景 4：未选择集合直接上传

**优先级**：P0
**异常点**：用户未在 Header 或 `/collections` 选择集合，直接在上传页拖拽文件。

**预期系统行为**：
- 拖拽区域可接收文件，但提交/上传时被拦截；
- Toast 提示“请先选择集合”；
- 文件不会真正进入上传流程。

**验收点**：
- `E2E-ERR-006`：未选集合时上传被阻止；
- `E2E-ERR-007`：提示文案准确指向“选择集合”操作。

### 场景 5：检索无结果

**优先级**：P1
**异常点**：查询词与任何文档不匹配。

**预期系统行为**：
- 页面展示 EmptyState：“无证据片段”；
- 提示检查查询词、集合、配置；
- 不展示虚假结果。

**验收点**：
- `E2E-ERR-008`：无结果时展示 EmptyState；
- `E2E-ERR-009`：页面不报错、不白屏。

## 4. 跨模块异常路径

以下场景验证各模块单点功能正常时，端到端链路是否仍可能断裂。

### 场景 6：驳回后重新上传

**优先级**：P0
**路径**：`/review` Reject → `/upload` 重传 → `/review` 新工单

**操作描述**：
1. 复核员在 `/review/[taskId]` 对工单点击 Reject；
2. 操作员回到 `/upload`，对同一文件或修正后的文件重新上传；
3. 新文件在 `/review` 生成新的待复核工单。

**预期系统行为**：
- Reject 后原工单状态变为“已拒绝”，不进入文档库；
- 重新上传后 `/upload` 文件列表新增记录，状态正常流转；
- `/review` 队列中出现新的待复核工单，与原工单独立；
- 新工单可正常进入 `/review/[taskId]` 审批。

**验收点**：
- `E2E-X-001`：Reject 后文档不入库；
- `E2E-X-002`：重新上传生成新工单；
- `E2E-X-003`：新工单状态为待复核，可再次审批；
- `E2E-X-004`：原工单与新工单互不干扰。

### 场景 7：退回修改后重新提交

**优先级**：P0
**路径**：`/review` Return → `/review/[taskId]` 编辑 Chunk → `/review` 重新提交

**操作描述**：
1. 复核员在 `/review/[taskId]` 点击 Return for revision；
2. 操作员/复核员在 Draft edits Tab 编辑 chunk；
3. 编辑完成后，工单状态可重新进入待复核或触发重新审批；
4. 复核员再次审批并通过。

**预期系统行为**：
- Return 后工单状态变为“已退回”；
- Draft edits 可正常编辑并提交到草稿层；
- 修改后工单可重新进入待复核状态；
- 再次 Approve 后文档正常入库。

**验收点**：
- `E2E-X-005`：Return 后状态正确；
- `E2E-X-006`：可编辑 chunk 并提交草稿；
- `E2E-X-007`：修改后可重新审批；
- `E2E-X-008`：再次 Approve 后文档入库。

### 场景 8：归档后检索不可见

**优先级**：P0
**路径**：`/documents/[docId]` Archive → `/retrieval` 验证不再命中

**操作描述**：
1. 管理员在 `/documents/[docId]` 对某已发布文档执行 Archive；
2. 操作员在 `/retrieval` 使用与该文档相关的查询词执行检索；
3. 验证该文档不再出现在检索结果中（或标记为已归档不可用）。

**预期系统行为**：
- Archive 后文档状态变为 Archived；
- 文档库中该文档状态同步更新；
- 检索结果中不再包含该文档的证据片段；
- 若检索仍命中，必须明确提示该文档已归档。

**验收点**：
- `E2E-X-009`：Archive 后状态更新；
- `E2E-X-010`：文档库状态同步；
- `E2E-X-011`：检索结果不再命中；
- `E2E-X-012`：若命中则提示已归档。

### 场景 9：撤回后状态回滚

**优先级**：P1
**路径**：`/documents/[docId]` Retract → `/review` 可重新复核

**操作描述**：
1. 管理员对已发布文档执行 Retract；
2. 验证文档从文档库移除或状态变为 Pending；
3. 原复核工单重新变为待复核状态（如业务如此设计）。

**预期系统行为**：
- Retract 后文档不再可被检索；
- 文档库状态更新；
- 关联工单状态回滚（若实现）。

**验收点**：
- `E2E-X-013`：Retract 后检索不可见；
- `E2E-X-014`：文档库状态更新。

### 场景 10：Token 过期后重新配置并重试

**优先级**：P0
**路径**：`/upload` 上传中 Token 过期 → `/settings` 重新配置 → `/upload` 重试

**操作描述**：
1. 用户已配置有效 token，在 `/upload` 上传文件；
2. 上传过程中 token 过期，API 返回 401；
3. 页面提示 token 过期并引导到 `/settings`；
4. 用户在 `/settings` 重新配置有效 token；
5. 返回 `/upload` 重新执行上传；
6. 上传成功并进入复核流程。

**预期系统行为**：
- Token 过期时 API 调用失败，页面不白屏；
- 明确提示“Token 已过期，请重新配置”；
- 提供入口跳转 `/settings`；
- 重新配置后 API 客户端使用新 token；
- 返回 `/upload` 后可继续上传，不需要刷新整个应用；
- 重新上传成功后状态正常流转。

**验收点**：
- `E2E-X-015`：Token 过期时给出明确提示；
- `E2E-X-016`：提示可跳转 `/settings`；
- `E2E-X-017`：重新配置 token 后 API 客户端生效；
- `E2E-X-018`：返回 `/upload` 可继续上传；
- `E2E-X-019`：重新上传后流程正常。
