# 00 - 全局可用性基线与验收规则

> **文档定位**：本文件是所有《Frontend Acceptance》验收文档的基线约束。Agent 在验收任何模块前，必须先确认本文件中的规则已满足。若某条基线被违反，无论业务功能是否正确，均视为该模块验收失败。
>
> **适用范围**：企业知识库工作台前端（`apps/web`），基于 Next.js 16 + React 19 + Tailwind v4。
>
> **关联需求**：`需求.md` 第 1 章（全局功能）、第 11 章（通用 UI 组件需求）、第 14.10 节（全局层面）。

---

## 1. 文档用途（D - Agent 知识库定位）

本文档不是某一条业务线的验收用例，而是**前置可用性契约**。它解决的是 Agent 在开发和自测时最常忽略的问题：

- 页面加载半天没有反馈，用户不知道系统在干什么；
- 按钮点了没反应，用户不知道操作是否成功；
- 出错后白屏或控制台报错，用户无法继续；
- 弹窗关不掉、Toast 不消失、空状态是空白页；
- 单点功能正常，但流程走到一半断掉。

**验收原则**：任何模块的验收，必须首先满足“用户能点、能走通、不卡死、不白屏”。本文件把这条原则具象化为可执行的检查项。

## 2. 通用可用性基线

### 2.1 加载与等待

| 基线编号 | 检查项 | 通过标准 | 优先级 |
|---|---|---|---|
| GLB-001 | 页面初始加载 | 任何页面从进入路由到首屏可交互，必须有 Skeleton 骨架屏或明确的加载指示（Spinner / Progress），禁止长时间白屏 | P0 |
| GLB-002 | 数据加载 | 列表、表格、详情等异步数据加载时，必须展示对应 Skeleton；数据返回后平滑替换，禁止闪烁或布局跳动 | P0 |
| GLB-003 | 加载超时 | 若加载超过 10 秒仍未返回，必须给出错误提示或重试入口，不能无限 loading | P0 |
| GLB-004 | 操作等待 | 提交、保存、审批等异步操作必须展示 loading 状态（按钮 disabled + spinner / 全局遮罩），禁止重复提交 | P0 |

### 2.2 交互反馈

| 基线编号 | 检查项 | 通过标准 | 优先级 |
|---|---|---|---|
| GLB-005 | 可点击元素反馈 | 按钮、链接、卡片等可交互元素必须有 hover / active / focus 状态，鼠标悬停时光标变为 `cursor-pointer` | P0 |
| GLB-006 | 禁用状态 | 未满足前置条件时（如未选集合、未填必填项），提交按钮必须明确 disabled，且用户能感知为何禁用 | P0 |
| GLB-007 | 操作成功反馈 | 创建、保存、删除、审批等操作成功后，必须通过 Toast（sonner）给出明确成功提示，文案语义正确 | P0 |
| GLB-008 | 操作失败反馈 | 任何失败必须通过 Toast 或 Alert 告知用户原因，不能静默失败或仅在控制台报错 | P0 |
| GLB-009 | 弹窗管理 | Dialog / Modal 必须有明确的关闭入口（X 按钮或取消按钮），点击遮罩层可关闭，Esc 键可关闭 | P0 |

### 2.3 错误与异常

| 基线编号 | 检查项 | 通过标准 | 优先级 |
|---|---|---|---|
| GLB-010 | 全局错误边界 | 任何渲染错误必须被 `<ErrorBoundary>` 捕获，展示友好错误页（含刷新按钮），不能白屏 | P0 |
| GLB-011 | 后端能力缺口 | 当 API 返回 HTTP 501 或未实现时，必须展示 `<BackendGap>` 组件，明确提示“功能尚未就绪”及缺失端点 | P0 |
| GLB-012 | 认证失败 | Token 过期或无效时，必须引导用户重新配置 token 或跳转设置页，不能无限报错 | P0 |
| GLB-013 | 网络错误 | 后端服务离线或网络超时时，必须给出离线提示或重试入口（如 `<OfflineToast>`），不能假死 | P1 |
| GLB-014 | 404 / 500 | 访问不存在路由或服务端错误时，必须展示 `not-found.tsx` / `error.tsx` 友好页面 | P0 |

### 2.4 空状态与数据缺失

| 基线编号 | 检查项 | 通过标准 | 优先级 |
|---|---|---|---|
| GLB-015 | 列表空状态 | 任何列表、表格、卡片网格在无数据时，必须展示 `<EmptyState>`（图标 + 标题 + 描述 + 可选操作按钮），禁止空白页 | P0 |
| GLB-016 | 筛选无结果 | 筛选后无匹配数据时，必须提示“无匹配结果”并提供“清除筛选”入口 | P0 |
| GLB-017 | 搜索空状态 | 搜索无结果时必须展示空状态，不能保持旧数据或显示错误 | P0 |

### 2.5 导航与状态

| 基线编号 | 检查项 | 通过标准 | 优先级 |
|---|---|---|---|
| GLB-018 | 当前页面高亮 | 侧边栏导航必须正确高亮当前所在页面 | P0 |
| GLB-019 | 页面切换动画 | 页面切换必须有淡入 + 上移动画，不能生硬跳转 | P1 |
| GLB-020 | 移动端抽屉 | 移动端侧边栏抽屉必须有点击遮罩关闭、汉堡/X 图标动画切换 | P1 |
| GLB-021 | 多标签页状态 | 当前选中的 Tab / 分段控制器必须有明确视觉指示 | P0 |

### 2.6 跨页面数据一致性

| 基线编号 | 检查项 | 通过标准 | 优先级 |
|---|---|---|---|
| GLB-022 | 跨页面状态同步 | 同一数据在不同页面（如上传页、文档库、复核详情）展示必须一致，禁止出现上传页显示“已入库”而文档库查不到的情况 | P0 |
| GLB-023 | 集合选择器同步 | Header 集合选择器、`/collections` 选中状态、`/upload` 前置校验三者必须实时同步 | P0 |
| GLB-024 | 任务状态与文档状态一致 | `/upload` 任务列表中的状态必须与 `/review` 工单状态、`/documents` 文档状态保持最终一致 | P0 |

## 3. 验收判定规则

当 Agent 执行验收时，必须按以下分类记录缺陷类型，避免把所有问题混为一谈。

### 3.1 功能可用缺陷（Functional Availability Defect）

**定义**：用户无法完成既定目标，或核心功能完全不可用。

**判定标准**（满足任意一条即属于此类）：

- 按钮/链接/输入框无法点击或点击无响应；
- 提交表单后数据未保存或操作未生效；
- 关键 API 调用失败且未给出反馈；
- 必填校验缺失，导致非法数据被提交；
- 未实现功能被当作已实现通过验收。

**验收结论**：**失败（Blocked）**。P0/P1 级功能可用缺陷阻塞发布。

### 3.2 视觉缺陷（Visual Defect）

**定义**：功能可完成，但界面呈现、样式、文案、布局异常，影响用户理解或操作信心。

**判定标准**（满足任意一条即属于此类）：

- 元素遮挡、重叠、错位、超出容器；
- 颜色/字体/间距与需求不符（如错误状态未用 destructive 红色）；
- 文案错误、缺失、截断、溢出；
- Badge/状态徽标显示错误或状态颜色不匹配；
- 动画缺失、生硬或方向错误。

**验收结论**：**失败（Visual）**。P0 级视觉缺陷（如关键按钮被遮挡）阻塞发布；P1/P2 级视觉缺陷可作为已知问题记录。

### 3.3 流程链路缺陷（Process Chain Defect）

**定义**：单点功能正常，但端到端业务流程无法连贯跑通；或状态流转不符合预期。

**判定标准**（满足任意一条即属于此类）：

- 上传成功后任务列表未及时更新状态；
- 复核通过后文档未进入已发布状态；
- 从前一步跳转后页面状态错误（如带着错误上下文进入下一步）；
- 状态徽标与实际后端状态不一致；
- 跨页面数据不同步（如集合选择器未同步）。

**验收结论**：**失败（Chain）**。P0/P1 级流程链路缺陷阻塞发布。

### 3.4 数据一致性缺陷（Data Consistency Defect）

**定义**：同一业务数据在多个页面、多个组件或前端与后端之间展示不一致，导致用户产生误判或无法继续操作。

**判定标准**（满足任意一条即属于此类）：

- 上传页显示“已入库”，但文档库查不到该文档；
- `/upload` 任务状态与 `/review` 工单状态不一致（超出合理延迟范围）；
- Header 集合选择器与 `/collections` 选中状态不同步；
- 审批决策后，复核详情页状态已更新，但文档库 Review 状态未同步；
- 检索结果中的 doc_id 在文档库中不存在或状态不一致；
- 统计面板数字与各页面明细汇总不一致。

**验收结论**：**失败（Data Consistency）**。P0 级数据一致性缺陷阻塞发布；P1 级需评估是否阻塞。

### 3.5 性能与稳定性缺陷（Performance & Stability Defect）

**定义**：功能正确，但响应慢、内存泄漏、频繁重渲染、卡死。

**判定标准**（满足任意一条即属于此类）：

- 页面首次加载超过 5 秒（无网络问题）；
- 列表滚动明显卡顿；
- 连续操作后浏览器卡死或内存持续增长；
- 轮询频率异常导致页面闪烁或性能下降。

**验收结论**：**失败（Performance）**。P0 级性能缺陷阻塞发布。

### 3.6 已知缺口 / 未实现功能

**定义**：需求文档明确要求，但当前代码未实现或仅部分实现。

**验收结论**：**失败（Not Implemented）**。必须明确标注“本版本不在验收范围”或“已知缺陷”，并记录需求来源。

## 4. 通用异常分支（E - 跨模块复用）

以下异常场景在每个模块文档中不再重复展开，但 Agent 验收任何模块时都必须检查：

### 4.1 环境 / 认证异常

| 异常编号 | 场景 | 预期系统行为 |
|---|---|---|
| ERR-AUTH-01 | 未配置 JWT Token | 访问需要认证的 API 时，提示用户前往 `/settings` 配置 token；受保护操作被禁用或引导配置 |
| ERR-AUTH-02 | Token 过期或无效 | API 返回 401 后，自动登出或提示重新配置 token，不能无限重试 |
| ERR-NET-01 | 后端服务全部离线 | 顶部健康状态灯全部变红，展示离线提示，页面进入降级状态 |
| ERR-NET-02 | 单个服务离线 | 对应服务健康灯变红，调用该服务的功能展示 BackendGap 或错误提示，其他功能不受影响 |
| ERR-NET-03 | 网络超时 | 操作超时后给出“请求超时，请重试”提示，提供重试按钮 |

### 4.2 业务前置异常

| 异常编号 | 场景 | 预期系统行为 |
|---|---|---|
| ERR-PRE-01 | 未选择知识库集合 | 上传、检索等依赖集合的功能必须禁用提交并提示“请先选择集合” |
| ERR-PRE-02 | 未配置权限范围 | 上传等功能必须禁用并提示配置权限范围 |
| ERR-PRE-03 | 无数据进入列表/详情 | 展示 EmptyState，不能报错或白屏 |

### 4.3 后端能力缺口异常

| 异常编号 | 场景 | 预期系统行为 |
|---|---|---|
| ERR-501-01 | API 返回 HTTP 501 | 展示 `<BackendGap>` 组件，明确提示缺失的端点或功能，不静默失败 |
| ERR-501-02 | 功能依赖的下游服务未就绪 | 对应操作按钮 disabled 或点击后展示 BackendGap，不能假死 |

### 4.4 操作异常

| 异常编号 | 场景 | 预期系统行为 |
|---|---|---|
| ERR-OP-01 | 重复提交 | 操作进行中按钮 disabled / loading，防止重复触发 |
| ERR-OP-02 | 无权限操作 | 管理员专属按钮对非管理员不可见或 disabled |
| ERR-OP-03 | 非法输入 | 表单校验即时反馈，禁止提交非法数据 |

## 5. 优先级定义

| 优先级 | 含义 | 验收结论 |
|---|---|---|
| **P0** | 核心功能 / 阻塞发布 | 失败即阻塞上线 |
| **P1** | 重要体验 / 主要业务路径 | 失败需评估是否阻塞 |
| **P2** | 增强功能 / 辅助路径 | 失败可作为已知问题 |
| **P3** | 未来优化 / 边缘场景 | 失败记录但不阻塞 |

## 6. 文档编写约定

后续模块文档统一遵循以下约定：

1. **编号规则**：模块代码 + 三位数字。如 `UPL-001` 表示 Upload 模块第 1 条用例。
2. **场景结构**：每条场景必须包含【正常流】和至少一条【异常/容错流】。
3. **验收点嵌入**：场景步骤中必须标注 `[B]` 单点验收点；关键 UI 步骤标注 `[C]` 视觉规格。
4. **基线引用**：每个模块文档开头必须声明“默认遵循 `00-全局可用性基线与验收规则.md`”。
5. **未实现标注**：任何需求文档中标注 ❌ 未实现 或 ⏳ 部分实现 的功能，必须在本模块文档“已知缺口”章节明确列出，验收结论为 **失败（Not Implemented）**。
6. **代码关联**：每份模块文档必须列出相关的页面组件路径、API 调用、E2E 测试文件。
