## packages/

### ragflow_runtime（177 文件，~100K 行）

**结论：不碰。**

这是 RAGFlow（InfiniFlow）的上游开源代码（Apache 2.0），不是团队自研。每份文件头有 `Copyright 2025 The InfiniFlow Authors. All Rights Reserved.`。

虽然表面看以下各组有重复模式，但实际逻辑不同：
- **rag_app/**（naive/qa/paper/manual/presentation/table/picture/audio/email/resume）— 各自处理完全不同的文档结构，不是同一套模式换参数
- **deepdoc/parser/** — 每格式（PDF/DOCX/PPT/XLSX/HTML/MD/JSON/TXT/EPUB）确实需要不同的解析逻辑
- **data_source/** — 来自 Onyx（MIT License），20+ connector 的模式虽相似但各有差异

重构它会偏离上游，导致无法合并补丁，ROI 极低。

### persistence（48 源文件，~30K 行）

**结论：不碰。**

40 个 repository 中仅约 10 个是纯 CRUD，其余有大量领域专属逻辑（状态机、乐观锁、双合约映射、复合主键等）。泛型 BaseRepository 最多省约 300 行模板代码（占总量的 ~1%），不值得为此引入抽象层。

### contracts（7 源文件，~2K 行）

**结论：不碰。**

`models.py` 1715 行含所有跨服务 Pydantic 模型。但拆文件不减少行数，pydantic 模型定义本身无可压缩空间。

### intake_runtime（20 文件，~3K 行）

**结论：不碰。**

团队自研代码，架构已干净（Protocol pattern + 纯函数 stage executor + 策略模式 converter），没有可砍的重复代码。真正短板是零测试覆盖，但那不是删行数的问题。

### documents（2 源文件，<1K 行）

**结论：不碰。**

过于单薄，没有可优化的东西。

## services/

### access（Java，31 源文件，~2K 行）

**结论：不碰。**

架构紧凑，MCP Streamable HTTP 支持，API Key 认证。已经够精简。

### admin（Python，41 源文件，~2.6K 行）

**结论：不碰。**

一致的模块模式（models → repository → routes → service），干净。

### indexing（Python，51 源文件，~6.5K 行）

**结论：基本不碰。**

`vendor/` 目录（6 文件，~300 行）与 `packages/ragflow_runtime/chunking/` 功能重叠，但需改 import 才能删除，且你已决定不碰 ragflow_runtime。其余代码正常。

### retrieval（Java，76 源文件，~6.5K 行）

**结论：不碰。**

最复杂的服务（20+ 包），混合检索管线清晰，FileProjection+JDBC 双实现是可测试性设计，不是冗余。

### intake-pipeline（Python，74 源文件，~12.9K 行）

**结论：不碰。**

7 个子服务构成摄入流水线编排，行数最多但结构合理。问题与 `intake_runtime` 一致：缺测试，不缺重构。

### workbench-api（Python，59 源文件，~7.7K 行）

**结论：不碰。**

25+ 包，但多数只有 1-2 个文件，无重复。

### smoke_tests（Python，3 文件，~1.3K 行）

**结论：不碰。**

纯测试套件，不讨论。

## 最终结论

**本项目经过多轮重构，业务代码（186K 行）已经没有明显的脂肪可砍。**

能确认的唯一重复是 `services/indexing/vendor/`（~300 行）与 `packages/ragflow_runtime/chunking/` 重叠。其余模块架构均合理、无大规模重复代码、不持有无意义的上游副本。

代码质量的下一个提升方向不是"删行数"，而是：
- **补测试覆盖**（尤其是 `intake_runtime`、`intake-pipeline` 子服务）
- **保持纪律**（不引入新重复、不过度抽象）
