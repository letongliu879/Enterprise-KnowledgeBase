## 二、`/upload` 死循环修复

### 2.1 根因

`src/hooks/use-local-storage.ts` 原实现：

```ts
const setValue = useCallback(
  (value: T | ((val: T) =>> T)) => {
    const valueToStore = value instanceof Function ? value(storedValue) : value;
    setStoredValue(valueToStore);
    window.localStorage.setItem(key, JSON.stringify(valueToStore));
  },
  [key, storedValue]
);
```

依赖数组包含 `storedValue`。每次调用 `setValue` 都会改变 `storedValue`，从而生成新的 `setValue` 引用。

`src/app/upload/page.tsx:347-349`：

```ts
useEffect(() => {
  setStoredBatch(snapshotFiles(files));
}, [files, setStoredBatch]);
```

`setStoredBatch` 是 `useLocalStorage` 返回的 `setValue`。由于引用不稳定，effect 每次渲染后都会重新执行，再次调用 `setStoredBatch`，再次触发渲染，形成死循环。

### 2.2 修复

改为函数式更新，移除 `storedValue` 依赖：

```ts
const setValue = useCallback(
  (value: T | ((val: T) => T)) => {
    const storage = getLocalStorage();
    try {
      setStoredValue((prev) => {
        const valueToStore = value instanceof Function ? value(prev) : value;
        if (storage) storage.setItem(key, JSON.stringify(valueToStore));
        return valueToStore;
      });
    } catch {
      // ignore
    }
  },
  [key]
);
```

同时增加 `getLocalStorage()` 守护函数，检测 localStorage 是否可用（防止 jsdom、隐私模式、禁用 localStorage 时抛错）。

### 2.3 测试

新增 `src/hooks/use-local-storage.test.tsx`，用组件模拟 `UploadPage` 的 `useEffect → setValue` 路径：

```tsx
function MirrorComponent() {
  const [stored, setStored] = useLocalStorage<number>("loop-test", 0);
  const [derived, setDerived] = useState(stored);

  useEffect(() => {
    setStored(derived);
  }, [derived, setStored]);

  return <button onClick={() => setDerived((d) => d + 1)}>count:{derived}</button>;
}
```

先看到 RED（点击后计数未更新、act 警告），修复后 GREEN。

## 三、根路由仪表盘修复

### 3.1 根因

`src/app/page.tsx` 原实现直接重定向：

```tsx
import { redirect } from "next/navigation";

export default function HomePage() {
  redirect("/upload");
}
```

导致 `/` 无法展示仪表盘。

### 3.2 修复

改为渲染已有仪表盘组件：

## 五、取消按钮测试修复

### 5.1 根因

`src/app/upload/page.tsx` 中取消按钮原实现用 Tooltip 包裹 Button：

```tsx
<TooltipTrigger>
  <Button ...><Square /></Button>
</TooltipTrigger>
```

`TooltipTrigger` 默认渲染为 `<button>`，内部又嵌套 shadcn `<Button>` 渲染的 `<button>`，产生无效嵌套，导致 `title="取消任务"` 没有落到真实 DOM 上，测试 `getByTitle` 失败。

### 5.2 修复

取消按钮改为和重试/删除按钮一致的原生 `title`：

```tsx
<Button
  variant="ghost"
  size="icon"
  title="取消任务"
  ...
>
  <Square className="h-3.5 w-3.5" />
</Button>
```

同时修复对应测试的 setup，让文件稳定在 `uploading` 状态（`uploadFileContent` pending、`listTasks` 返回 uploading）。

## 六、复测新增修复：`/upload` 状态映射崩溃

### 6.1 现象

使用 agent-browser 对修复后的 `/upload` 进行复测时，页面仍出现客户端 500：

```text
TypeError: Cannot read properties of undefined (reading 'icon')
  at src/app/upload/page.tsx:1294:43
```

### 6.2 根因

后端 `/workbench/tasks` 返回的任务状态包含 `completed`（以及可能返回的 `publishing`、`unknown`），但前端：

- `UploadStatus` 类型未包含这些状态；
- `KNOWN_UPLOAD_STATUSES` 未包含这些状态；
- `getStatusConfig` / `getStatusLabel` 的 `switch` 分支未覆盖这些状态。

当 `recentTasks` 中存在 `completed` 等状态时，`getStatusConfig` 返回 `undefined`，访问 `config.icon` 即崩溃。

### 6.3 修复

| 文件 | 修改 |
|------|------|
| `src/lib/api/types.ts` | `UploadStatus` 增加 `"completed"`、`"publishing"`、`"unknown"` |
| `src/app/upload/page.tsx` | `KNOWN_UPLOAD_STATUSES` 增加上述状态；`getStatusConfig` 增加对应分支与 `default` 兜底；`getStatusLabel` 增加对应标签与 `??` 兜底 |

### 6.4 验证

- 单元测试：`npx vitest run` 相关 4 个文件 **57 tests passed**。
- agent-browser 复测 `/upload`：页面正常渲染，不再崩溃。

## 八、browser-use 复测新修复

### 9.1 Documents 等页面 hydration 错误 / Issues badge

**现象**：复测截图中 `/documents` 页面左下角仍有 Next.js Issues badge；console 报：

```text
In HTML, <button> cannot be a descendant of <button>.
This will cause a hydration error.
```

**根因**：`@base-ui/react` 的 `TooltipTrigger` / `DropdownMenuTrigger` 默认渲染一个 `<button>`，而项目里大量代码直接把 `<Button>` 作为子元素塞进去，导致 SSR/CSR 生成嵌套 button，触发 hydration 错误。

**修复**：

1. 给 `src/components/ui/tooltip.tsx` 和 `src/components/ui/dropdown-menu.tsx` 的 Trigger 增加 `asChild` 支持，内部使用 `render` prop 把子元素作为触发器根元素。
2. 把所有 `<TooltipTrigger><Button ... /></TooltipTrigger>` 和 `<DropdownMenuTrigger><Button ... /></DropdownMenuTrigger>` 改为带 `asChild`。
3. `src/app/documents/page.tsx` 的 `DocumentActionMenu` 也改为 `asChild`。

**验证**：

- agent-browser 复测所有页面 console 无 error，snapshot 中不再出现 `Open issues overlay`。
- 前端单元测试：5 个相关文件 97 tests passed。

### 9.2 `/retrieval` 端到端检索失败

**现象**：选择集合和配置后点击“检索上下文”，前端无结果；后端返回 500 或空 `knowledge_context`。

**根因链**：

1. `services/workbench-api/.env` 缺少 `ACCESS_INTERNAL_API_KEY`，导致 `AccessClient` 未带 `X-API-Key`，Access 服务返回 `ACC_UNAUTHENTICATED`。
2. `workbench-api` 发往 `/v1/retrieve` 的 payload 包含 `tenant_id`/`application_profile_id`/`user_id`/`max_results`/`budget_policy` 等 Access `ExternalRetrieveRequest` 不认识的字段，且缺少必填的 `collection_scope`。
3. `WorkbenchQueryRunModel` 没有 `trace_id` 列，但 `retrieval.py` 把 `trace_id` 写进了 `repo.create(...)`，导致 SQLAlchemy `TypeError`。
4. Access 返回的是 `KnowledgeContext` 本身（含 `evidence_items`），而 `workbench-api` 按 `{knowledge_context: {...}}` 取值，导致返回前端的 `evidence_items` 为空。

**修复**：

- 在 `services/workbench-api/.env` 增加 `ACCESS_INTERNAL_API_KEY`。
- 在 Access 服务投影中同步该 API Key（`/internal/api-key-projections/sync`）。
- 在 admin/retrieval runtime 中补充 `ret_smoke_01` profile，使其在前端可选。
- 修改 `services/workbench-api/src/workbench_api/commands/retrieval.py`：
  - `access_payload` 只保留 `query`、`collection_scope`、`token_budget`、`retrieval_profile_id`、`debug`。
  - 从 `repo.create(...)` 中移除 `trace_id`。
  - `knowledge_context = result.get("knowledge_context") or result`，兼容 Access 直接返回的对象。

**验证**：

- `curl` 调 `POST /workbench/retrieve` 返回正常 `knowledge_context` 与 `evidence_items`。
- agent-browser 在 `/retrieval` 选择 `Test Collection` + `Smoke Retrieval Profile`，输入 `test` 后成功返回证据片段，截图显示“检索到的证据片段 检索耗时 516ms · 后端 410ms”。
