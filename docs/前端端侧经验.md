# 前端端侧 E2E 验收经验

> 记录一次 Knowledge Workbench 浏览器端 E2E  walkthrough 的全过程、踩坑与经验。
> 时间：2026-06-13  
> 工具：agent-browser 0.27.2 + Next.js 16.2.6 前端 + FastAPI 后端

## 一、任务背景

目标：模拟真实用户在 Chromium 中完整操作企业知识库工作台，按顺序访问 9 个页面/交互点，截图并输出验收报告。

原始要求：
- 前端：http://localhost:3000
- 后端 API：http://127.0.0.1:18083
- 管理员 token 已配置在 localStorage
- Chromium 视窗 1440×900
- 截图保存到 `test-results/screenshots/{page-name}.png`

最终产出：
- `test-results/screenshots/*.png`（9 张页面截图）
- `test-results/report.md`（验收报告）
- 本文档：`docs/前端端侧经验.md`

## 二、环境启动与基础检查

### 2.1 启动前端

前端位于 `apps/web`，使用 Next.js：

```bash
cd apps/web
npm run dev
```

默认监听 `http://localhost:3000`。如果 3000 端口已被占用，Next.js 会提示是否换端口；本次任务要求固定 3000，因此需先确认端口空闲。

### 2.2 检查后端的真正状态

不要只看端口是否通，要看 API 是否能正常返回业务数据：

```bash
# 健康检查

curl http://127.0.0.1:18083/workbench/health

# 认证接口（用于确认 token 是否有效）

curl -H "Authorization: Bearer $TOKEN" http://127.0.0.1:18083/workbench/dashboard

# 集合列表
curl -H "Authorization: Bearer $TOKEN" http://127.0.0.1:18083/workbench/collections
```

坑点：某些接口（如 `/workbench/health`、`/workbench/auth/me`）不访问数据库，即使数据库未启动也会返回 200。只有真正查询 DB 的接口（如 `/workbench/dashboard`）才会暴露数据库连接问题。

### 2.3 后端数据库配置坑

`workbench-api` 的 `get_session()` 在未设置 `DATABASE_URL` 时，默认连接 PostgreSQL（`postgresql://reality:reality@127.0.0.1:5432/reality_rag`）。如果没有 PostgreSQL，任何 DB 查询都会返回 500。

修复方式：

在 `services/workbench-api/.env` 和 `services/admin/.env` 中分别设置：

```env
# services/admin/.env
DATABASE_URL=sqlite:///admin.db
ADMIN_JWT_SECRET=change-me-in-production

# services/workbench-api/.env
DATABASE_URL=sqlite:///admin.db
JWT_SECRET=change-me-in-production
```

注意：两个服务的 `JWT_SECRET` 必须一致，否则 token 在一个服务能验过、在另一个服务会报 `Invalid token`。

## 三、认证与 Token 踩坑

### 3.1 localStorage 中的 token 可能是假的

任务说明“管理员 token 已配置在 localStorage”，但实际情况可能是：
- token 已过期
- token 格式正确但签名 secret 与后端不一致
- token 是之前某次调试遗留的，后端已重启或换了 secret

前端 `useAuthGuard` 只检查 token 是否为三段式 JWT：

```ts
const hasJwtToken = !!demoToken && demoToken.split(".").length === 3;
```

因此前端会显示“已认证”，但实际调用 API 时全部 401/500。

### 3.2 如何验证 token 真的有效

不要只在前端看，要用 `curl` 直接调后端：

```bash
TOKEN="你的JWT"
curl -s -H "Authorization: Bearer $TOKEN" http://127.0.0.1:18083/workbench/collections
```

如果返回 `{"detail":{"error_code":"UNAUTHORIZED","message":"Invalid token"}}`，说明 token 与当前后端 secret 不匹配。

### 3.3 如何注入/更新 token

前端使用 zustand persist，localStorage key 为 `ekb-workbench-store`，字段为 `state.demoToken`。

可以用 agent-browser 直接设置：

```bash
agent-browser storage local set ekb-workbench-store '{"state":{"demoToken":"你的JWT",...},"version":0}' --session workbench-session
agent-browser reload --session workbench-session
```

注意：
- 注入后必须 `reload`，否则 zustand 仍使用内存中的旧值。
- token 会同步到 cookie（`ekb_workbench_token`），用于图片/文件下载等场景。

### 3.4 自己生成 token 的注意事项

如果后端 secret 是 `change-me-in-production`，可以用 Node 生成：

```bash
node -e "
const crypto = require('crypto');
const b64 = o => Buffer.from(JSON.stringify(o)).toString('base64url');
const header = { alg: 'HS256', typ: 'JWT' };
const payload = {
  sub: 'admin-1',
  email: 'admin@example.com',
  roles: ['platform_admin', 'knowledge_admin'],
  tenant_id: 'default',
  allowed_collections: ['*'],
  exp: Math.floor(Date.now()/1000) + 86400
};
const secret = 'change-me-in-production';
const signing = b64(header) + '.' + b64(payload);
const sig = crypto.createHmac('sha256', secret).update(signing).digest('base64url');
console.log(signing + '.' + sig);
"
```

坑点：网上流传的 token 不一定匹配当前运行的后端 secret，必须亲自验证。

## 四、使用 agent-browser 的详细过程

### 4.1 安装与版本

```bash
npm i -g agent-browser
agent-browser install
agent-browser --version  # 0.27.2
```

### 4.2 核心工作流

agent-browser 是单会话、命令式的 CDP 浏览器自动化工具。基本流程：

```bash
# 1. 打开页面
agent-browser open <url> --viewport 1440x900 --name workbench-session

# 2. 查看页面（获取 @eN 引用）
agent-browser snapshot -i --session workbench-session

# 3. 操作元素
agent-browser click @e3 --session workbench-session
agent-browser fill @e4 "text" --session workbench-session
agent-browser select @e5 "option-value" --session workbench-session

# 4. 等待
agent-browser wait @e6 --session workbench-session
agent-browser wait --load networkidle --session workbench-session
agent-browser wait --text "成功" --session workbench-session

# 5. 截图
agent-browser screenshot /path/to/file.png --session workbench-session

# 6. 结束
agent-browser close --all
```

### 4.3 常用命令速查

| 命令 | 作用 |
|------|------|
| `open <url>` | 导航到 URL |
| `reload` | 刷新当前页 |
| `snapshot -i` | 输出可交互元素的 accessibility tree（带 `@eN` 引用） |
| `get url` / `get title` | 获取当前 URL/标题 |
| `click @eN` / `click "selector"` | 点击元素 |
| `fill @eN "text"` | 清空并输入 |
| `type @eN "text"` | 追加输入 |
| `press Enter` / `press Meta+k` | 按键 |
| `select @eN "value"` | 下拉选择 |
| `find text "跳过" click` | 语义定位并点击 |
| `wait @eN` / `wait --load networkidle` / `wait 2000` | 等待 |
| `screenshot path.png` | 截图 |
| `storage local set key value` | 设置 localStorage |
| `storage local get key` | 读取 localStorage |
| `eval "js code"` | 执行 JS |

### 4.4 daemon 不稳定时的处理

agent-browser 有时会出现：

```text
Invalid response: EOF while parsing a value at line 1 column 0 (after 5 retries - daemon may be busy or unresponsive)
```

处理办法：

```bash
# 关闭所有 session
agent-browser close --all

# 等待几秒，让 daemon 重启
sleep 5

# 重新打开（可带 --debug 看详细日志）
agent-browser --debug open http://localhost:3000 --viewport 1440x900 --name workbench-session
```

如果反复出现，可以在每条命令之间加 `sleep 2~5`。

### 4.5 session 名称的使用

所有命令都要加 `--session workbench-session`，否则操作的是 `default` session。常见问题：
- 在 A session 里 inject token，B session 没生效。
- 截图发现页面是空白，因为操作的是另一个 session。

### 4.6 截图路径

Windows Git Bash 下，agent-browser 对相对路径支持不稳定，建议用绝对路径：

```bash
agent-browser screenshot /c/AI/My-Project/Enterprise-KnowledgeBase/test-results/screenshots/review.png --session workbench-session
```

### 4.7 处理弹窗/引导

页面有首次引导弹窗时，会遮挡内容。每次导航后都可能重新出现。处理模式：

```bash
agent-browser find text "跳过" click --session workbench-session || true
```

用 `|| true` 避免“元素不存在”导致命令失败。

## 五、遇到的端侧 Bug

### 5.1 `/upload` 页面死循环导致 500

现象：访问 `/upload` 显示客户端 500 错误页。

控制台错误：

```text
Maximum update depth exceeded. This can happen when a component calls setState inside useEffect...
  at useLocalStorage.useCallback[setValue] (src/hooks/use-local-storage.ts:20)
  at UploadPage.useEffect (src/app/upload/page.tsx:348)
```

根因：`useLocalStorage` 的 `setValue` 在 `useEffect` 中被调用，且依赖变化导致每次渲染都触发 setState，形成死循环。

影响：
- `/upload` 无法使用。
- `/` 重定向到 `/upload`，所以首页仪表盘也无法访问。

### 5.2 `/` 重定向到 `/upload`，没有仪表盘路由

`src/app/page.tsx` 直接 `redirect("/upload")`，导致任务要求的“首页仪表盘”无法访问。仪表盘组件 `features/dashboard/dashboard-page.tsx` 存在但未被使用。

### 5.3 Hydration mismatch

首次加载时 Next.js 报 hydration 不匹配，涉及 `AppShell` 和 `RootLayout`。虽然页面能继续渲染，但可能是引导弹窗或状态同步问题的诱因之一。

### 5.4 检索配置缺失

`/retrieval` 页面表单渲染正常，但“检索配置”下拉为空，无法执行检索。需要为集合创建并发布 retrieval profile。

### 5.5 复核详情 workspace 降级

手工注入的工单未关联 source_file / parse_snapshot，导致复核详情页提示：

```text
Workspace is partially degraded: ticket_detail, agent_review.
```

## 六、数据准备经验

为了让页面有内容，需要准备：

1. **集合（collection）**：通过 admin API 创建。
   ```bash
   curl -X POST -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
     -d '{"collection_id":"coll-qa-1","tenant_id":"default","name":"QA Collection 1"}' \
     http://127.0.0.1:18084/admin/collections
   ```

2. **文档投影**：直接写 `workbench_document_projection` 表。
3. **工单投影**：直接写 `workbench_ticket_projection` 表。
4. **任务投影**：直接写 `workbench_task_projection` 表。

坑点：
- 表结构可能和直觉不同，插入前先用 `PRAGMA table_info(table_name)` 确认字段。
- 不要假设 `task_id` 字段存在；实际表用的是 `projection_id` 做主键。
- 布尔字段 `is_stale` 必须给值，不能省略。

## 七、验收流程建议

1. **先通后验**：先确认前后端都 healthy、token 有效，再开始 UI 操作。
2. **一页一确认**：每个页面加载后先 `snapshot -i` 或截图，确认关键元素存在再继续。
3. **token 必验证**：用 `curl` 验证 token 能调通后端，不要只相信前端的“已认证”状态。
4. **截图要命名清晰**：`{page-name}.png`，不要混用 `debug.png`。
5. **遇到 500 先抓日志**：看浏览器控制台或 `apps/web` 的 dev server 日志，定位是前端还是后端问题。
6. **引导弹窗要处理**：每次 `open` 新 URL 后都可能弹出 onboarding tour，统一先跳过。
7. **失败也要截图**：任务要求“遇到任何错误立即截图并记录”，所以 `/upload` 的 500 页也要作为 `upload.png` 保存。

## 九、总结

本次 E2E 验收最大的教训是：**不要相信“token 已配置”和“服务已启动”的假设**，必须逐项验证：

1. 端口通 ≠ 服务健康。
2. 前端显示“已认证” ≠ 后端认可 token。
3. 健康接口 200 ≠ 数据库可用。
4. 组件存在 ≠ 路由正确。

agent-browser 适合这种单会话、逐步点击的浏览器验收，但需要注意 daemon 稳定性和 session 一致性。遇到 500 时，优先查看 Next.js dev server 的终端输出，通常比截图更能定位根因。

最终问题清单：
- [x] 修复 `/upload` 页面的 `Maximum update depth exceeded` 死循环。
- [x] 决定首页 `/` 是渲染仪表盘还是保留重定向（已改为渲染仪表盘）。
- [x] 为测试集合创建发布检索配置（`ret_qa_01`、`ret_smoke_01`）。
- [ ] 补充完整测试数据（source_file / parse_snapshot）——仍待 intake 流水线真实跑通后补充。

复测新增教训：
5. 截图无 Issues badge 才能真正说明页面“干净”； hydration 错误必须修到根因，不能只修一处。
6. 后端服务“端口通”之后，还要验证真实业务链路（如一次完整 retrieve）才能算可用。
7. Workbench → Access → Retrieval 的端到端检索需要 API Key 投影、Retrieval Profile 投影、集合索引三者同时就绪，缺一不可。
